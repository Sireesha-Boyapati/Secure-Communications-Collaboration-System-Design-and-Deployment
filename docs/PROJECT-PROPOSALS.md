# Project Proposals — Secure Communications / Collaboration System

**Module:** B9IS103 — Computer Systems Security 2026  
**Group size:** 4  
**Cloud platform:** AWS  
**Database:** MongoDB Atlas  
**Purpose:** Lecturer review and team selection of one idea to implement

---

## Shared security model (all proposals)

All four ideas use the same cryptographic foundation. Differences are **use case**, **identity verification**, and **UI/workflow**.

### Trust assumptions

| We trust | We do **not** trust |
|----------|---------------------|
| User's browser / Web Crypto API | Our AWS application server |
| TLS (HTTPS / WSS) in transit | MongoDB administrators or DB leaks |
| Standard algorithms (ECDH, AES-GCM, SHA-256) | Any third party with server/database access |

### Key exchange without meeting (common approach)

1. Each user generates an **ECDH key pair** in the browser on first login.
2. Only the **public key** is uploaded to MongoDB.
3. When User A messages User B, A's browser fetches B's public key and runs **ECDH** to derive a shared secret.
4. Shared secret → **HKDF** → **AES-256-GCM** session key.
5. Plaintext is encrypted **in the browser** before sending to the server.
6. Server stores and relays **ciphertext only**.
7. Users verify a **key fingerprint** (SHA-256 of public key, shown as 6 groups) on a second channel (Zoom, phone) to detect server key substitution.

### What the server cannot do (if implemented correctly)

- Read message plaintext
- Derive session keys from stored data alone (no private keys on server)

### Known limitations (documented in all proposals)

- Malicious server could swap public keys → mitigated by fingerprint verification
- Compromised user device exposes local private keys
- DDoS can affect availability (not confidentiality)
- Metadata (who talks to whom, when) may still be visible to server

# Idea 1: StudySafe

## Tagline

**Encrypted real-time group chat for student teams**

## Daily-life use case

Students coordinate every day on deadlines, lab work, exam times, file passwords, and group assignment splits. StudySafe provides **invite-only encrypted rooms** so even a compromised AWS server or MongoDB cannot read messages.

**Example scenario:** Four lab partners create room `B9IS103-Team`. Alice sends *"Section 3 draft is in shared drive — password in next message"*. Only room members decrypt it. Lecturer demo: show MongoDB contains random ciphertext.

## Target users

- Project teams (3–20 members)
- Study groups
- Thesis / dissertation pairs

## Functional requirements (MVP)

| ID | Requirement |
|----|-------------|
| FR1 | Register with email and verify via OTP |
| FR2 | Create an encrypted chat room |
| FR3 | Invite members via email link or 6-character join code |
| FR4 | Send and receive real-time messages (WebSocket) |
| FR5 | Display key fingerprint for out-of-band verification |
| FR6 | Message history loaded from MongoDB (ciphertext only on server) |

## Non-functional requirements

| ID | Requirement |
|----|-------------|
| NFR1 | Message delivery latency < 2 seconds on campus Wi-Fi |
| NFR2 | Support at least 20 members per room |
| NFR3 | HTTPS/WSS only; no plaintext HTTP |
| NFR4 | Mobile-responsive web UI |

## Security requirements

| ID | Requirement |
|----|-------------|
| SR1 | Private keys never leave the client browser |
| SR2 | Server stores public keys and ciphertext only |
| SR3 | AES-256-GCM with unique nonce per message |
| SR4 | Identity verified via email OTP before key registration |
| SR5 | Key fingerprint displayed for MITM detection |
| SR6 | JWT authentication; short-lived access tokens |
| SR7 | Rate limiting on login and registration |

## Identity verification

**Primary:** Email OTP (university or personal email).  
**Secondary:** Key fingerprint compared on Zoom / phone during first session.  
**Optional extension:** Restrict registration to `@*.ac.uk` or college domain.

## Tech stack

| Layer | Technology |
|-------|------------|
| Frontend | React 18 + TypeScript |
| Client crypto | Web Crypto API (ECDH P-256, AES-GCM, SHA-256) |
| Backend | Python 3.12 + FastAPI |
| Real-time | WebSockets (`fastapi` + `websockets`) |
| Database | MongoDB Atlas (AWS region) |
| Auth | JWT + email OTP (AWS SES or SendGrid) |
| API docs | OpenAPI (auto-generated by FastAPI) |
| AWS compute | EC2 (t3.small) or ECS Fargate |
| AWS networking | Application Load Balancer + ACM certificate |
| AWS CDN | CloudFront (optional) |
| Secrets | AWS Secrets Manager |
| Monitoring | CloudWatch logs + basic alarms |
| CI/CD | GitHub Actions → deploy to EC2/ECS |
| Repo | GitHub (this repository) |

## AWS architecture

```
Browser (encrypt/decrypt)
    → CloudFront / ALB (HTTPS, WSS)
    → EC2 FastAPI + WebSocket
    → MongoDB Atlas (users, rooms, ciphertext)
    → AWS SES (OTP emails)
```

## MongoDB collections

- `users` — email, publicKey, fingerprint, verified, createdAt
- `rooms` — name, memberIds, inviteCode, createdBy
- `messages` — roomId, fromUserId, ciphertext, nonce, iv, createdAt

## Team split (4 members)

| Member | Responsibility |
|--------|----------------|
| 1 | React UI — chat, rooms, invite flow |
| 2 | FastAPI REST + WebSocket relay |
| 3 | Email OTP auth, JWT, AWS deployment |
| 4 | README, threat model, diagrams, report, MoM |

## Complexity & timeline

| | Rating |
|--|--------|
| Implementation difficulty | **Medium** (recommended) |
| Time to MVP | ~2 weeks |
| Demo impact | **High** — team uses it for own assignment |
| Identity story | Clear (email + fingerprint) |

## Why select this idea

- Immediately useful for **this group of 4** as first real users
- Strong **collaboration** angle (group rooms, not just 1:1)
- Easiest to explain in presentation with live two-browser demo
- Maps cleanly to assignment language (maliciously curious server)

# Idea 2: FlatShare LockChat

## Tagline

**Encrypted coordination for housemates and shared living**

## Daily-life use case

People in shared houses discuss rent splits, utility bills, landlord visits, Wi-Fi passwords, and guest access daily on WhatsApp. FlatShare LockChat replaces that with **E2E encrypted flat rooms** where the relay cannot read sensitive household information.

**Example scenario:** Four housemates in room `Apt-42-B`. Bob sends *"Landlord inspection Tuesday 10am"* and later *"New Wi-Fi password: …"* — encrypted end-to-end.

## Target users

- Student house shares (2–6 people)
- Young professional flatshares
- Short-term rental groups

## Functional requirements (MVP)

| ID | Requirement |
|----|-------------|
| FR1 | Create a "Flat" (household room) with name and address nickname |
| FR2 | Invite flatmates via email link or QR join code |
| FR3 | Real-time encrypted group chat per flat |
| FR4 | Pinned "House info" encrypted note (optional MVP+) |
| FR5 | Key fingerprint verification between flatmates |

## Non-functional requirements

| ID | Requirement |
|----|-------------|
| NFR1 | Usable on phone browsers |
| NFR2 | One user can belong to multiple flats (e.g. moved house) |
| NFR3 | 99% uptime during demo period |

## Security requirements

Same SR1–SR7 as Idea 1, plus:

| ID | Requirement |
|----|-------------|
| SR8 | Invite links expire after 48 hours |
| SR9 | Only flat members can read room ciphertext |

## Identity verification

**Primary:** Email OTP.  
**Secondary:** Join code shared in person at the flat (physical channel).  
**Optional:** Phone SMS OTP via AWS SNS for flat invite confirmation.

## Tech stack

| Layer | Technology |
|-------|------------|
| Frontend | React 18 + TypeScript + Tailwind CSS |
| Client crypto | Web Crypto API |
| Backend | Node.js 20 + Express + `ws` (WebSocket) |
| Database | MongoDB Atlas |
| Auth | JWT + email OTP (AWS SES) |
| AWS compute | EC2 or Elastic Beanstalk |
| AWS networking | ALB + ACM |
| Secrets | AWS Secrets Manager |
| CI/CD | GitHub Actions |
| Repo | GitHub |

## MongoDB collections

- `users` — email, publicKey, fingerprint
- `flats` — name, memberIds, inviteCode, expiresAt
- `messages` — flatId, fromUserId, ciphertext, nonce, createdAt
- `pinnedNotes` — flatId, ciphertext (optional)

## Team split (4 members)

| Member | Responsibility |
|--------|----------------|
| 1 | React UI — flat setup, chat, invite QR |
| 2 | Express API + WebSocket |
| 3 | Auth + SMS/email + AWS deploy |
| 4 | Threat model + domestic use-case report + demo script |

## Complexity & timeline

| | Rating |
|--|--------|
| Implementation difficulty | **Medium** |
| Time to MVP | ~2–2.5 weeks |
| Demo impact | **High** — relatable non-technical story |
| Identity story | Email + in-person join code |

## Why select this idea

- Strong **real-world narrative** outside academia
- Physical invite code adds interesting **security discussion**
- Good contrast with Idea 1 while sharing same crypto core

# Idea 3: LocalTrust Chat

## Tagline

**Encrypted 1:1 chat for local buy/sell negotiations**

## Daily-life use case

When buying or selling on Facebook Marketplace, DoneDeal, or campus noticeboards, users share **price, pickup address, phone number, and payment details**. Platforms and relay servers can read these. LocalTrust provides **1:1 encrypted negotiation** between buyer and seller who have never met.

**Example scenario:** Seller lists a textbook. Buyer opens LocalTrust, finds seller by email/username, verifies fingerprint on a quick phone call, then sends *"Can you do €25? Pickup at library Monday 4pm"* — encrypted.

## Target users

- Students buying/selling books, electronics, furniture
- Anyone doing local peer-to-peer deals
- Tutoring / freelance rate negotiation

## Functional requirements (MVP)

| ID | Requirement |
|----|-------------|
| FR1 | Register with email OTP |
| FR2 | Search user by verified email or username |
| FR3 | Start 1:1 encrypted conversation |
| FR4 | Real-time message delivery (WebSocket) |
| FR5 | "Safety checklist" UI before sharing address (fingerprint verified?) |
| FR6 | Block / report user (metadata only on server) |

## Non-functional requirements

| ID | Requirement |
|----|-------------|
| NFR1 | 1:1 chat only in MVP (scope control) |
| NFR2 | Conversation list with last-message timestamp (no preview plaintext on server) |

## Security requirements

Same SR1–SR7 as Idea 1, plus:

| ID | Requirement |
|----|-------------|
| SR8 | Warn user if fingerprint not verified before sending address-like content |
| SR9 | No plaintext message preview in push notifications |

## Identity verification

**Primary:** Email OTP.  
**Secondary:** Key fingerprint verified by phone before sharing sensitive data.  
**Optional extension:** Link listing ID from marketplace in first message for context.

## Tech stack

| Layer | Technology |
|-------|------------|
| Frontend | React 18 + TypeScript |
| Client crypto | Web Crypto API + `libsignal-protocol` simplified (ECDH only for MVP) |
| Backend | Python FastAPI |
| Real-time | WebSocket |
| Database | MongoDB Atlas |
| Auth | JWT + email OTP |
| AWS | EC2 + ALB + ACM + CloudWatch |
| CI/CD | GitHub Actions |
| Repo | GitHub |

## MongoDB collections

- `users` — email, username, publicKey, fingerprint
- `conversations` — participantIds (2 users), createdAt
- `messages` — conversationId, fromUserId, ciphertext, nonce, createdAt

## Team split (4 members)

| Member | Responsibility |
|--------|----------------|
| 1 | React UI — user search, 1:1 chat, safety checklist |
| 2 | FastAPI + WebSocket + conversation logic |
| 3 | Auth + AWS deployment |
| 4 | Report — marketplace threat model, attack scenarios |

## Complexity & timeline

| | Rating |
|--|--------|
| Implementation difficulty | **Medium** (1:1 simpler than groups) |
| Time to MVP | ~2 weeks |
| Demo impact | **Very high** — everyone understands buy/sell privacy |
| Identity story | Email + phone fingerprint before address share |

## Why select this idea

- Clear **privacy motivation** for people who have never met
- 1:1 scope is **smaller** than group chat — good if time is tight
- Excellent **vulnerability analysis** section (fake seller, MITM, phishing)

# Idea 4: GitTrust Collab

## Tagline

**GitHub-verified identity with encrypted project messaging**

## Daily-life use case

Developers and students collaborate on GitHub but discuss sensitive details (API keys locations, unreleased features, grading concerns) on Discord or email. GitTrust links **GitHub identity** to encrypted messaging so you know you are talking to the owner of a specific GitHub account.

**Example scenario:** Alice finds Bob via GitHub username `@bobdev`. App verifies Bob controls that account via OAuth, fetches Bob's published public key from a GitHub Gist, then opens an encrypted channel for project coordination.

## Target users

- Software project teams
- Open-source collaborators
- Students sharing code review feedback privately

## Functional requirements (MVP)

| ID | Requirement |
|----|-------------|
| FR1 | Sign in with GitHub OAuth |
| FR2 | Publish public key to GitHub Gist (one-time setup wizard) |
| FR3 | Find users by GitHub username |
| FR4 | 1:1 encrypted real-time chat |
| FR5 | Optional encrypted markdown note per conversation |
| FR6 | Display GitHub avatar + username alongside fingerprint |

## Non-functional requirements

| ID | Requirement |
|----|-------------|
| NFR1 | OAuth token stored encrypted server-side; minimal GitHub scopes |
| NFR2 | Works when GitHub is reachable for setup only |

## Security requirements

Same SR1–SR7 as Idea 1, plus:

| ID | Requirement |
|----|-------------|
| SR8 | Public key must match key fetched from user's GitHub Gist at conversation start |
| SR9 | Alert if GitHub key changes (possible account compromise) |
| SR10 | GitHub OAuth scope limited to `read:user` only |

## Identity verification

**Primary:** GitHub OAuth — proves control of `@username`.  
**Secondary:** Public signing/encryption key hosted in user-owned GitHub Gist.  
**Tertiary:** Key fingerprint verification on video call.

This directly matches the assignment example: *"Social media based identity, with profiles being used to certify identity"* and *"PKI with keying information posted on HTTPS sites"*.

## Tech stack

| Layer | Technology |
|-------|------------|
| Frontend | React 18 + TypeScript |
| Client crypto | Web Crypto API |
| Backend | Python FastAPI |
| Real-time | WebSocket |
| Database | MongoDB Atlas |
| Identity | GitHub OAuth 2.0 |
| Key distribution | GitHub Gist API (public key JSON) |
| AWS | EC2 + ALB + ACM |
| Secrets | AWS Secrets Manager (GitHub client secret, JWT secret) |
| CI/CD | GitHub Actions |
| Repo | GitHub |

## MongoDB collections

- `users` — githubId, username, avatarUrl, publicKey, gistUrl, fingerprint
- `conversations` — participantIds, createdAt
- `messages` — conversationId, ciphertext, nonce, createdAt
- `notes` — conversationId, ciphertext (optional)

## Team split (4 members)

| Member | Responsibility |
|--------|----------------|
| 1 | React UI + GitHub OAuth flow |
| 2 | FastAPI + Gist key fetch + WebSocket |
| 3 | AWS deployment + secrets management |
| 4 | Identity verification report + strongest threat-model write-up |

## Complexity & timeline

| | Rating |
|--|--------|
| Implementation difficulty | **Medium–Hard** (OAuth + Gist integration) |
| Time to MVP | ~2.5–3 weeks |
| Demo impact | **Very high** for technical audience |
| Identity story | **Strongest alignment with brief** |

## Why select this idea

- Most distinctive **identity verification** story in the class
- Combines social identity + HTTPS-hosted public keys as brief describes
- Great for presentation Q&A about account hijacking and key rotation
